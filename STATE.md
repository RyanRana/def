# Project State: DEF — Dynamic Engagement Framework

---

## 1. Goal

- **Lightweight agent framework** that adapts to **any web game**. Using **attention-based reinforcement learning**, it **modifies live games** in real time (speed, gravity, obstacles, rewards, difficulty, forgiveness) to **maximize dopamine and engagement**.
- **Vitals** (heart rate, breathing rate, expression) from **in-browser inference** only: webcam + rPPG. No native SDKs.
- **Full mod suite** per game generated from game code analysis and applied automatically by the agent; game source code is never edited.

---

## 2. Architecture (Web Only)

### 2.1 Flow

1. User opens a game (e.g. `/games/snake.html`). The **server injects** `engage-bootstrap.js` before `</body>`; game source files are **not edited**.
2. Bootstrap loads: **game_state_\<game\>.js** (state capture) → **game_mods_\<game\>.js** (mod registry) → **engage-adapter.js** → **camera-consent.js**.
3. After consent: **engage-tracker.js** → **inference-vitals.js** → **engage-logger.js** → **engage-tracker-ui.js** → **engage-stimulus.js** → **engage-rl.js**.
4. The **agent** runs as a lightweight layer: the tracker builds the 18-vector every tick; the **stimulus** layer (Thompson sampling + attention) produces `__engageModPlan`; the **attention-based RL engine** classifies engagement, selects policies via UCB1, and **modifies the live game** through `__gameMods` to maximize dopamine and engagement.

### 2.3 Vector (18 fields — true time series)

`[heartRate, breathingRate, expressionIdx, sessionTimeSec, score, deaths, tapCount, activeTouchMs, scoreDelta, hrDelta, brDelta, deathsDelta, touchRatio, deathRate60s, keyVelocity, keyRhythm, faceMovement, emotionValence]` — plus optional `gameState` from `getEngageGameState()`.

### 2.4 Game state (per-game, for time-series)

Generated from `scripts/game_profiles/<game>.json`. Each tick captures all fields via `window.getEngageGameState()`:

- **Snake (11):** head_x, head_y, dir_dx, dir_dy, length, max_cells, apple_x, apple_y, grid, canvas_w, canvas_h.
- **Dino (10):** distance_ran, speed, crashed, playing, started, obstacle_count, jump_velocity, trex_y, is_jumping, is_ducking.
- **Bird (7):** score, lost, paused, pipe_count, fly_area_height, bird_top, bird_left.

### 2.5 Game mods (per-game, RL-controlled)

Generated from `scripts/game_profiles/<game>.json`. Each mod has key, label, category, default, min, max, step, get(), set(). Categories: speed, gravity, obstacle, reward, difficulty, forgiveness.

**Engagement allowlist:** Each profile can define `engagement_mod_keys` (array of mod keys). Only these mods are used by the stimulus layer and RL engine for automatic adjustments; other mods remain in the registry for the overlay and manual use. This avoids poor choices (e.g. canvas size) and keeps auto-modding to high-impact levers (speed, apples, gravity, forgiveness). `game_mods_<game>.js` sets `window.__engageModKeys` from the profile when present.

- **Snake:** game_frame_skip (speed), death_forgiveness, apple_attract_radius (reward). Others blocked or hand-tuned.
- **Dino:** initial_game_speed, game_acceleration, gravity, initial_jump_velocity, obstacle_gap_coefficient, gameover_clear_time, achievement_distance (max_speed blocked).
- **Bird:** gravity_acceleration, jump_height, pipe_spawn_interval_loops (terminal_velocity blocked).

### 2.6 Attention-Based RL: Live Game Modification for Max Engagement

The agent classifies the player’s engagement state from rolling signal windows and applies game-specific mods to the **live game** to maximize dopamine and engagement.

**Engagement States** (classified from 18-vector rolling window):
- **FLOW**: Steady input, moderate scoring, stable vitals. Optimal — maintain.
- **BORED**: Low/declining input rate, flat score, low taps. → Speed up, cluster rewards, add variability.
- **FRUSTRATED**: High death rate, recent deaths, declining engagement. → Ease obstacles, lower gravity, slow down, increase forgiveness.
- **EXCITED**: High scoring, high input, HR volatility. → Ride the wave, slight speed boost, more rewards.
- **DISENGAGED**: Low input + high deaths — about to quit. → Dramatic rescue: big rewards, ease everything.

**Policy Selection** (UCB1 Contextual Bandit):
- Each engagement state has 2-3 candidate policies (e.g. `bored_speed_up`, `bored_reward_cluster`, `bored_variability`).
- UCB1 selects which policy to try, balancing exploration vs exploitation.
- After each decision cycle, the bandit observes the change in engagement score and updates arm rewards.
- Over a session, the bandit learns which policies work for this player/game combination.

**Mod Application**:
- Policies output category deltas (e.g. `{speed: +0.08, reward: +0.12, obstacle: -0.05}`).
- Deltas are scaled by urgency (consecutive bored/frustrated states amplify adjustments).
- Applied to all mods in the matching category via `__gameMods[key].set(newValue)`.
- Every application is logged: `[RL Mod] speed/game_speed: 6.00 → 7.20 (policy=bored_speed_up, state=bored, score=0.35)`.

**Key design: lightweight agent, any web game**. The agent reads whatever mods are in `__gameMods` and applies by category to modify the live game. Adding a new game requires only a game profile (via Gemini or manually).

## 3. Run

- **Server:** `npm run serve` (or `node server.js`). Port 8765, bind 0.0.0.0.
- **Games:** `http://localhost:8765/games/snake.html` (or bird/dino).
- **Regenerate profiles:** `GEMINI_API_KEY=... python analyze_games.py` — produces game_profiles, game_state_*.js, game_mods_*.js.

## 4. Out of scope (web-only)

- No Presage SDK, Expo, Capacitor, or native apps. Vitals are in-browser inference only.
